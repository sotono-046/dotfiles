# SEO Checklist (112 項目)

サイト全体または個別ページの SEO を、ドメイン設計からマークアップ、運用までの 112 項目で監査・改善するための skill。**監査依頼時は該当節を順に確認**、**実装依頼時は対象節のみ参照** する。

## いつ使うか

- 新規サイト構築時に SEO 観点の初期設計を固めたい
- 既存サイトの SEO 監査を依頼された
- 個別ページ・テンプレートの SEO レビューを依頼された
- リニューアル / ドメイン移管時の SEO 影響を確認したい

使わない場面:

- 特定のテクニカル SEO ツール (Search Console, GA4) の使い方だけを知りたい → 該当ツールのドキュメントを参照
- コンテンツの中身 (文章品質・トピック設計) のみ → 本 skill は構造面が主、コンテンツ戦略は別軸

## 使い方

1. 対象を確認する: サイト全体 / 単一ページ / テンプレート / リニューアル
2. 該当節をスキャンし、項目ごとに **○ / × / 不明 / N/A** を判定する
3. × 項目を **影響度 (高 / 中 / 低)** で並べ替える
4. 修正案を「項目番号」「現状」「あるべき姿」「修正方法」の 4 点で提示する
5. ユーザーが許可した修正のみ適用する (canonical / robots.txt / リダイレクト等は影響範囲が大きいため必ず確認)

## チェックリスト

### 1. ドメイン・URL (1〜10)

1. 独自ドメインを使う
2. ドメイン名の長さは、あまり長くならないようにする
3. 将来的に多言語展開がある場合、`.jp` は使わない
4. サブドメインの利用は非推奨 (サブディレクトリを優先)
5. URL (スラッグ) は、簡潔でわかりやすいアルファベット表記
6. URL は不変で、変わらないものにする
7. URL は静的なもので、パラメーターにより区別されるものは極力避ける。トラッキング系パラメータ (`utm_*`, `ref`, `fbclid` 等) は、ページ内容を変えないので **canonical でパラメータ無し URL に正規化** する (項目 97 と連動)
8. 文字コードは UTF-8
9. 同一 URL で、PC / スマホ両対応のレスポンシブサイト
10. SSL (https) 対応

### 2. ローカル / ビジネス情報 (11)

11. Google ビジネスプロフィールに登録する (該当業種のみ)

### 3. キーワード設計 (12〜19)

12. 対象キーワードの検索ボリュームは、十分多いものを使う
13. 同じようなキーワード (検索意図が同じ、検索結果がほぼ同じ) はグループ化して、一つとして扱う
14. キーワードの種類 (購入・行動につながるクエリと情報収集のクエリ) を考慮して、ページに割り当てる
15. 全国対象ではない場合は、地名を追加して確認
16. メインで狙うキーワードを一番最初に決める (社内相談必須)
17. 競合調査は、メインで狙うキーワードで検索した上位の類似サイトを調べる
18. 競合とは違う軸で勝てる要素をコンセプトにする
19. 主要なすべてのページにキーワードを割り当てる

### 4. サイト構造・情報設計 (20〜25)

20. サイト全体を親子孫の上下階層がある構成にする
21. トップページはサイト全体の目次として設計
22. グローバルナビを設置し、重要なページへの導線を確保する
23. フッターナビを設置し、主要なページへの導線を確保する
24. すべてのページがオリジナルの内容にする
25. 類似ページ、重複ページが多数になるような設計をしない

### 5. ページ内コンテンツ配置 (26〜33)

26. 狙うキーワードを極力、上部 (ファーストビュー内が理想) に設置
27. コンテンツの下部、サイドバー、フッターなどから関連ページへのリンクを設置
28. 子供のページへのリンクは、コンテンツとして配置する
29. 長いページは最初に目次などを設置して内部リンクを設置
30. 極端に文字数の少ないページが出来ないようにする (目安: 本文 **300 字未満は警告**、**200 字未満は ×**。ただしカテゴリトップなど一覧主体のページは N/A 判定可)
31. タイトルとリンクのみのページは、極力作らない
32. カテゴリページ・一覧ページにも、リンク以外でオリジナリティを確保する
33. 1 ページ = 1 テーマにする

### 6. キーワードのコンテンツ内利用 (34〜40)

34. 狙うキーワードをコンテンツに含める
35. 狙うキーワードが複合キーワードである場合は、できる限り近接させる
36. 狙うキーワードの表記ゆれがある場合は、すべてのパターンを自然に入れ込む
37. 狙うキーワードを乱用しない
38. キーワードの出現率を計算しない (Google は密度を直接評価しない)
39. 関連キーワード、メインキーワードを意識して入れる
40. 狙うキーワードに対してユーザーのニーズを満たす内容を入れる

### 7. 見出し (h1〜h4) と title (41〜49)

41. `h1 > h2 > h3` の構造を想定してライティングする
42. `h2`、`h3` は、それぞれの親に対して 3 つ以上存在するようにする (できれば)
43. `h1` と `title` タグは、ほぼ同じ内容にする
44. `h1` と `title` タグには、狙うキーワードを前方に配置
45. `h1` と `title` タグの文字数は、長くしすぎない (title は 30 字前後目安)
46. `h1` と `title` タグは、ページ内容を適切に表すもの
47. `h1` はページに 1 回を基本
48. `h2` 〜 `h4` を利用して構造化する
49. メインコンテンツ以外に `h` タグを使わない

### 8. HTML マークアップ (50〜57)

50. HTML5 を極力取り入れる (`article`, `section`, `nav`, `aside`, `header`, `footer`)
51. `strong` を乱用しない
52. `em` は基本使わない
53. 表は `table`
54. リストは `ul`, `ol`
55. 引用は `blockquote`
56. インラインフレーム (`iframe`) は使わない (埋め込みが必須の場合を除く)
57. `description` (meta description) を適切に記述する

### 9. ページネーション / 画像 (58〜63)

58. `rel="prev"` / `rel="next"` は不要 (Google は 2019 年にサポート終了)
59. メインビジュアルも含めて、テキストを画像にしない
60. 画像の `alt` は、内容を表すものにする
61. 説明の画像などは、`caption` をつけるか `figure` 構成にする
62. レスポンシブイメージにする (`srcset`, `sizes`)
63. ブログなどの場合、遅延読み込みにする (`loading="lazy"`)

### 10. リンク (64〜69)

64. アンカーテキストは、可能な限りリンク先の内容を表す内容にする
65. 可能なら、アンカーテキストに狙うキーワードをいれる
66. 画像でリンクする場合は、`img` の `alt` を表す内容がリンク先になるようにする
67. リンクは静的リンク (`a href=""`)、javascript・動的生成にしない
68. 広告、対価 (お金以外も含む) を貰って掲載するリンクは、`rel="sponsored"`
69. 意味がないリンクは設置しない

### 11. 表示・UX (70〜81)

70. HTML で記載の内容は、初期状態ですべて見れるようにする (CSS で隠さない、JS 描画に頼らない)
71. 派手な動作をつけない
72. スマートフォン対応 (toC は、モバイルファーストデザインを基本)
73. PC とスマホで、表示内容を同じにする
74. `h` タグ構造と文字の大きさは比例させる
75. `h` タグ構造と目立ちやすさは比例させる
76. デザインや配置のために、`table` を使わない
77. リンクはリンクとわかるデザインにする
78. ボタンはボタンとわかるデザインにする
79. 画面範囲外に要素を飛ばさない
80. 隠しテキストになるような記述をしない
81. インタースティシャル (画面全体を覆う広告) は禁止

### 12. クロール支援 (82〜86)

82. `sitemap.xml` を生成する
83. パンくずリストを設置する
84. 構造化データを設置する (JSON-LD 推奨)
85. 無限スクロールを利用する場合は、ページネーションでもアクセス可能にする
86. コメントを開放する場合、承認制にする

### 13. パフォーマンス (87〜88)

87. 1 ページの出力ソース、CSS など関連ソースのサイズを抑える
88. ページ速度の最適化、ページエクスペリエンス (Core Web Vitals) で良好になるようにする

### 14. URL 統合・リダイレクト (89〜91)

89. http / https、www あり / なしの統合をする (301 で正規化)
90. `index.html`、`index.htm`、`index.php` などの統合をする
91. リニューアル時、過去のサイトの全 URL にアクセス可能にする (301 リダイレクトで対応)

### 15. 重複・noindex 設計 (92〜102)

92. クロールが不要なページは生成しない
93. 類似ページは、生成されないようにする
94. **【通常サイト】** 検索結果ページは、`noindex`
95. `robots.txt` は必要な場合のみ生成
96. `robots.txt` の記述は矛盾しないようにする
97. `rel="canonical"` は、基本は、自分自身の初期状態 URL
98. 一覧ページの 1 ページ目に canonical を戻さない
99. 言語が違う場合は、canonical ではなく、`rel="alternate"` (`hreflang`)
100. リダイレクトコードは適切に利用する (恒久 = 301、一時 = 302)
101. **【EC / DB サイト】** 検索結果をインデックスさせるかどうかを厳密に設計する
102. **【EC / DB サイト】** 検索結果 0 件、在庫なし、販売終了の扱いを厳密に設計する

### 16. サーバー応答・配信 (103〜106)

103. HTTP コードを適切に返す (404 / 410 / 503 等)
104. ユーザー向けと GoogleBot 向けに違う内容を返さない (cloaking 禁止)
105. キャッシュの設定は適切にする
106. javascript による動的生成コンテンツ (ajax) は極力使わない

### 17. 運用・保守 (107〜112)

107. リンク切れをチェックして、修正する
108. 古くなったコンテンツは最新情報へ更新する
109. Search Console を確認、エラー (赤字) があれば対応する
110. リンクを購入しない、お金を払ってリンクをしてもらわない
111. 外部サイトへ掲載した内容と同じものを自社サイトに乗せる場合は、`noindex` をしてもらう
112. ウイルス感染、スパム汚染などを定期的に確認 (Search Console、サーバーのメールチェック)

## 監査時の出力フォーマット

監査結果は次の表で提示する:

```
## SEO 監査結果

| # | 項目 | 判定 | 影響度 | 現状 | 修正案 |
|---|------|------|--------|------|--------|
| 10 | SSL 対応 | × | 高 | http のみ | Let's Encrypt で https 化 + 301 リダイレクト |
| 60 | 画像 alt | × | 中 | 全画像 alt 空 | 主要画像 N 件に内容説明の alt を追加 |
...

## 重要な発見
- <影響度 高 の項目を 3-5 件箇条書き>

## 次のアクション
1. <優先度順に並べた修正タスク>
```

## 影響度の目安

- **高**: クロール / インデックスを直接阻害する (9, 10, 89, 91, 94, 97, 100, 103, 104, 109)
- **中**: ランキング要因に直接効く (43, 44, 57, 60, 82, 84, 88)
- **低**: 補助的・微調整 (38, 42, 51, 52)

優先度が分からない項目は「中」を初期値とし、サイト規模・業種で調整する。

## EC / 大規模サイト向け判断基準 (項目 101, 102 の補助)

項目 101 (検索結果インデックス可否) と 102 (在庫切れ / 販売終了の扱い) は「厳密に設計」とだけ述べているため、典型ケースの decision tree を示す。

### ファセット / 検索結果 URL (101)

`/search?q=...&color=red&size=L` のような **無限増殖型ファセット URL** は、まず **生成抑止 → クロール制御 → canonical 集約** の順で考える:

| 状況 | 推奨対処 | 関連項目 |
|---|---|---|
| ユーザーにも必要ない組み合わせ | そもそも URL を生成しない | 7, 92, 93 |
| 検索結果ページ全般 (通常サイト) | `noindex` | 94 |
| 主要ファセット (色・サイズ等) で需要があり、独自コンテンツがある | インデックス対象、自分自身に canonical | 97, 101 |
| パラメータの順序違いだけで同一内容 | canonical で代表 URL に集約 | 97 |
| クロール予算節約のためそもそも辿らせたくない | `robots.txt` でブロック (ただし既にインデックス済みは noindex 経由で外す) | 95, 96 |

**NG パターン**: ファセット URL の canonical を一律トップに向ける → 項目 97 / 98 違反、Google は canonical 無視。

### 在庫切れ / 販売終了商品 (102)

| 状況 | HTTP / メタ | 関連項目 |
|---|---|---|
| 一時的な在庫切れ・再入荷予定あり | `200 OK` のまま「在庫切れ」表示 | 102, 103 |
| 恒久的に販売終了、代替商品なし | `410 Gone` (または `404`) | 103 |
| 恒久的に販売終了、明確な代替商品あり | 代替商品ページへ `301` | 100, 103 |
| カテゴリ自体が廃止 | カテゴリトップへ `301` | 100 |

**NG パターン**: 数万ページの販売終了商品を一律トップへ `301` / canonical → 項目 102 / 104 違反、Google にはソフト 404 扱いされる。

## 注意

- 本リストは 2026 年時点の一般的なガイドラインを反映。Google のアルゴリズム変更で一部項目 (58 の `rel="prev/next"` のように) は陳腐化することがある。**監査時に最新の Google Search Central ドキュメントを参照** すること。
- 項目 16 (メインキーワード決定) は社内相談必須。エージェントが単独で決めない。
- 項目 100 (リダイレクト) や 95-97 (robots / canonical) は誤設定で順位が消えるリスクがあるため、適用前に必ずユーザー確認を取る。
