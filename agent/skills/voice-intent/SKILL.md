# Voice Intent Normalization

音声入力由来のプロンプトから正しい意図を取り出すための正規化手順。`agent/AGENTS.md` の「コミュニケーション」節の詳細版。

## 1. 誤変換・音写の検出と照合

以下のパターンを誤変換の疑いとして検出する。

- カタカナの技術用語で、コードベース内の識別子と音が似ているが表記が異なるもの（例: 「バリファイゲート」「グリーンレート」）
- 英単語の音写ゆらぎ（例: 「アナリシス」「アナリシス関数」）
- 言い直し・自己訂正を含む発話（「〜じゃなくて〜」「あー違う、〜」）

検出したら、対象リポジトリで `grep -ri` 等により近い識別子・定数名・ドキュメント上の用語を探し、最も一致するものへ変換して解釈する。一致が複数候補あり確信が持てない場合のみ、着手前に「対象は◯◯と解釈しました」と一行で復唱してから進める。質問だけして手を止めない。

## 2. 指示語の解決

「これ」「ここ」「この画面」「なんか上の方が〜」等の指示語は、次の優先順で対象を解決する。

1. 直前に共有された画像・スクリーンショット
2. 現在開いているブラウザの URL・ページ内容
3. 直前の会話で言及されたファイル・コンポーネント
4. 上記で決まらない場合は、最も妥当な解釈を一文で明示してから作業に着手する

## 3. 複数タスクの分解

1 メッセージ内に複数の依頼が並記されている場合（「〜して。あと〜も。ついでに〜」）は、内部チェックリストに分解し、すべての項目に対応する。抜けがあると再度同じ指示が繰り返される原因になるため、完了報告では依頼項目ごとに対応結果を書く。

## 4. 承認・権限フレーズの判定

- `OK` 単独、`いいよ`、`していいよ` 等は同意 + 続行の合図として扱う
- `マージしていい` 等は明示的な権限付与であり、以後の追加確認なしに該当操作まで進める
- 感想形の指摘（「〜な気がする」「若干違う?」）は修正依頼として扱う
- 「これって何を根拠にしてる?」等の問いは説明要求であり、先に根拠を提示し修正は行わない
