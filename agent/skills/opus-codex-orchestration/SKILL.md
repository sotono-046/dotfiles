# Opus × Codex オーケストレーション

大きな開発を **人間 ←→ Opus ←→ Codex** の多重下請け構造で進める。Opus（このセッション）は司令塔として意図と本質を握り、実装の詳細は Codex に委譲する。

## 設計思想

それぞれの得意・不得意を前提に役割を固定する。

| 役割 | 担当 | 得意 | 不得意（だから任せない） |
| ---- | ---- | ---- | ------------------------ |
| 司令塔 | **Opus**（自分） | 意図を汲む・本質をつかむ・ツッコミ役のレビュー | コードを書くとミスで手戻りが多い |
| オペレーター | **Codex** | プラン立案（トークン効率よく広く深く短く）・実装を漏れなく実行 | 詳細にとらわれて本質を見落とす |

核心となる経験則:

- **プランは Codex に書かせ、Opus がレビューする。** Opus にプランを書かせると詳細に踏み込みすぎる。Opus はツッコミ役に徹したほうがうまくいく。
- **実装も Codex に書かせる。** Opus が直接コードを書くのは最小限にする。
- **Opus は本質チェックに徹する。** Codex が細部に没頭して目的を見失っていないかを監視する。

## 前提

- Codex は次のいずれかで起動する。
  - **MCP**: `mcp__codex__codex`（新規セッション開始）/ `mcp__codex__codex-reply`（同一セッション継続）。マルチターンで文脈を引き継ぎたいときはこちら。
  - **CLI**: `timeout 1800 codex exec '<プロンプト>'`（非対話・単発）。長い調査や実装で早期終了しないよう 30 分タイムアウトを既定にする。
- Opus はコードベースの理解に Serena（`get_symbols_overview` / `find_symbol` など）を優先し、ファイル全体の読み込みを避ける。
- 重い実装は `run_in_background` で投げ、`TaskOutput` で回収してもよい。

## ワークフロー

### フェーズ0: 意図の把握（Opus）

人間の依頼から「本当に解きたいこと」を抽出する。曖昧な点は Codex に投げる前に人間へ確認する。Codex に渡すのは「何を・なぜ」であって「どう書くか」ではない。

**確認の閾値**: ゴール・認可・データ整合・後戻りしにくい設計判断に関わる曖昧さは必ず人間に確認する。表示順・命名など可逆な詳細は仮置きし、フェーズ5の報告で「仮置きした」と明示する（確認で止まりすぎない）。

成果物: Codex へのプラン依頼文（ゴール・制約・対象範囲・非ゴールを明記）。

### フェーズ1: プラン立案（Codex）

Codex にプランだけを書かせる。**この段階では実装させない。**

```
mcp__codex__codex を呼び出し:
  prompt: |
    以下のゴールに対する実装プランを作成してください。実装はまだ行わないでください。
    プランは「変更対象ファイル / 手順 / 影響範囲 / リスク」を簡潔に列挙してください。

    ## ゴール
    <Opus が整理した意図>

    ## 制約・非ゴール
    <制約>
```

Codex はトークン効率よく広く・深く・短くプランを出す。これを Opus が受け取る。

### フェーズ2: プランレビュー（Opus = ツッコミ役）

Opus はプランを **批判的に** レビューする。詳細な書き直しはせず、本質的なツッコミに徹する。

レビュー観点:

- ゴールから外れていないか（本質の見落とし）
- 抜け漏れ（考慮されていないケース・依存・副作用）
- 過剰設計・スコープ膨張
- リスクの過小評価

問題があれば `mcp__codex__codex-reply` で同一セッションに差し戻し、プランを修正させる。**Opus が自分でプランを書き直さない。** 指摘 → Codex 修正 → 再レビューを、本質的な懸念が消えるまで繰り返す。

**収束しないときの打ち切り**: 同一観点を指摘した差し戻しを2通送っても解消しない場合、Codex を責める前に依頼文（ゴール・非ゴールの明示）が不十分と疑い、依頼文を強化して1回再試行する。それでも解消しなければ人間にエスカレーションする（「Codex がこの点を直せない／判断が必要」と要判断事項として報告）。無限に往復しない。

> より厳密なレビューが必要なら `plan-digger` サブエージェントに引き継いでよい。

### フェーズ3: 実装（Codex）

承認したプランに沿って Codex に実装させる。

```
mcp__codex__codex-reply を呼び出し（同一セッション継続）:
  prompt: |
    承認されたプランに従って実装してください。
    プランの範囲を超える変更は行わず、逸脱する場合は理由を述べてから行ってください。
```

長時間になる実装は `timeout 1800 codex exec` をバックグラウンドで起動して回収する手もある。

### フェーズ4: 本質レビュー & 検証（Opus）

Codex の実装差分を Opus がレビューする。ここでも Opus は「書き直す人」ではなく「本質を確認する人」。

- `git diff` で変更を確認する
- 当初のゴールを満たしているか（細部に没頭して目的を外していないか）
- プラン外の予期せぬ変更がないか
- 明らかなバグ・抜けがないか

問題があればフェーズ3に差し戻し（`codex-reply`）。品質・型・リントの機械的検証は `quality-gainner` サブエージェントに任せてもよい。移行・リファクタなどで「挙動が変わっていないか」を確かめる振る舞い等価性の検証（旧新の出力をゴールデン比較するなど）は `simulation-quality-loop` skill に委譲してもよい。

### フェーズ5: 人間への報告（Opus）

人間にはプロセスではなく結論を簡潔に報告する。手戻りした点・残課題・確認してほしい判断を明示する。

## 役割の境界（厳守）

- ❌ Opus がプランや実装コードを最初から書く
- ✅ Opus は意図整理・ツッコミ・本質検証・人間との対話に専念する
- ❌ Codex に「どう実装すべきか」を細かく指示しすぎる（本質を見失わせる）
- ✅ Codex には「何を・なぜ」を渡し、漏れない実行を任せる
- ⚠️ ただし些末な修正や Codex 起動コストが見合わない小タスクは、Opus が直接やってよい。本スキルは「大きな開発」向け。
  - **切り分け基準**: 「Codex のプラン→レビュー→実装ループのオーバーヘッドより作業実体が小さい」かつ「本質を外すリスクが低い機械的変更（typo・文言・1ファイルの局所修正など）」の両方を満たすなら Opus 直対応。どちらか怪しければオーケストレーションに乗せる。

## Codex 起動の使い分け

| 状況 | 使う |
| ---- | ---- |
| プラン → レビュー → 実装と文脈を引き継ぎたい | `mcp__codex__codex` → `mcp__codex__codex-reply` |
| 単発の調査・プラン生成 | `timeout 1800 codex exec '<prompt>'` |
| 長時間の実装・全テスト実行 | `timeout 1800 codex exec` を `run_in_background` で起動し `TaskOutput` で回収 |

**競合時の優先規則**: 「長時間の実装」と「差し戻しでの文脈引き継ぎ」が同時に必要な場合（例: バックグラウンド実装の差分をレビューして再実装させたい）、文脈引き継ぎを優先して MCP セッション（`codex-reply`）を使う。文脈が薄く再起動しても困らないなら `codex exec` + background で構わない。判断軸は「前回の文脈を失うと差し戻しが成立しないか」。

ハイブリッドも可: 指揮系統（プラン承認・差し戻し）を MCP セッションに置きつつ、純粋に時間のかかる単発実行だけ `codex exec` + background に切り出す。その場合 background のプロンプトに差し戻しに必要な文脈（承認済みプラン・前提仕様）を自己完結で埋め込み、文脈喪失を補償する。

## チェックリスト

### プラン依頼前

- [ ] 人間の意図・ゴール・非ゴールを Opus が整理した
- [ ] 曖昧な要件は人間に確認済み
- [ ] Codex には「何を・なぜ」を渡し「どう書くか」は渡していない

### プランレビュー時

- [ ] 本質（ゴール一致）を最優先で確認した
- [ ] 抜け漏れ・過剰設計・リスク過小評価をツッコんだ
- [ ] Opus 自身でプランを書き直していない（差し戻しで修正させた）

### 実装レビュー時

- [ ] `git diff` で全変更を確認した
- [ ] ゴールを満たし、本質を外していないか確認した
- [ ] プラン外の変更がないか確認した
- [ ] 機械的品質検証（型・リント）を実施または `quality-gainner` に委譲した

### 完了時

- [ ] 人間に結論・残課題・要判断事項を簡潔に報告した
