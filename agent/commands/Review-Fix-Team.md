# レビュー → 修正オーケストレーションチーム

## 目的

PR の差分を `/code-review`（Skill: `code-review`）基準でレビューし、その指摘を**ファイル単位の非競合グループに束ねてグループ数分の修正サブエージェントを並列起動**する。**各サブエージェントは担当指摘を自分でコード修正し、完了後にコミットする。** これを**再度 `/code-review` をかけて「対応すべき指摘がゼロ／OK」になるまでループ**する。発動したエージェント（自分）は司令塔として、レビューの起動・指摘のグルーピング・並列修正の統括・ループ判定に専念する。グループが1つで修正量が小さい場合は、サブエージェントを起動せず**自分で直接修正**してよい。

## 前提（PR が必須）

レビュー基準として用いる `/code-review` は **GitHub PR を前提**とし、指摘を PR にコメントする設計である。本コマンドもこれに合わせ、**対象 PR がある場合のみ動作**する。

- `$ARGUMENTS` で対象 PR を指定する（PR 番号 / PR URL / PR ブランチ）。
- **`$ARGUMENTS` が空（PR 番号未指定）の場合は、現在のブランチから PR を自動検出する**。`gh pr view --json number,state,headRefName,url`（引数なし）で現ブランチに紐づく PR を取得し、見つかればその PR を対象として続行する。ユーザーへの確認は不要。
- 現ブランチからも PR が見つからない場合に限り、レビューを起動せずここで停止し、ユーザーに「PR を指定／作成してください」と伝える。ローカルの未コミット差分だけを対象に本コマンドを動かさない。

> 前提スキル: **フェーズ0で PR OK と判定しループに入る直前**に `task-orchestration` スキル（並列実行の原則：「同一ファイルの同時編集禁止」「依存タスクは順次」）を読み、そのまま適用する。コミットは `git-ops` skill に従う。コード編集は `agent/AGENTS.md` の Serena 優先方針に従う。並列修正の共通方針も同ファイルの「並列修正」節と整合する。

## 役割分担

| 役割                               | 担当                                 | 仕事                                                                                                                                                                                                             |
| ---------------------------------- | ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 司令塔 / レビュー統括              | **発動エージェント**（自分）         | `/code-review` を起動して指摘を取得し、**ファイル単位の非競合グループ**に束ねる。グループ数分の修正サブエージェントを並列起動し、修正後に再レビューしてループ判定する。1グループ・軽量なら自分で直接修正してよい |
| 修正サブエージェント（複数・並列） | **サブエージェント**（グループ数分） | 割り当てられた**1グループの指摘だけ**を扱う。**自分でコードを修正**し、担当外ファイルには触れず、完了後に**自分で1コミット**する                                                                                 |

## 並列化の核心

**ファイル非競合かつ依存のない指摘を、1サブエージェントに直列押し込みしない。** 指摘を非競合グループに分け、**グループが2つ以上**あるときはグループ数だけサブエージェントを**1回のメッセージ内で同時に**起動する（複数 Agent 呼び出しを並列発行）。各サブエージェントは担当分を直接修正・コミットする。N グループが N 並列で同時に修正・コミットされ、直列実行の待ち時間を排除する。

> **依存束ね優先**: 強い依存（フェーズ2 ルール4）で同一グループに束ねた指摘は、グループ内で順次対応する。ファイルが重ならなくても依存で1グループになった場合は並列分割しない。

## ループ概要

```
[PR前提チェック] → (NGなら停止)
   ↓
┌──────────── ループ（最大3周）────────────┐
│ フェーズ1: /code-review でレビュー（自分）   │
│   → 対応すべき指摘がゼロ/OK なら ⇒ 完了で抜ける │
│ フェーズ2: 指摘のグルーピング（自分）         │
│ フェーズ3: 修正（2グループ以上→並列サブエージェント、│
│   1グループ・軽量→自分で直接修正・コミット）     │
│ フェーズ4: 各修正のコミット確認（自分）        │
└─────────── 周回して再びフェーズ1へ ──────────┘
   ↓
フェーズ5: 報告（OK到達 or 3周到達でエスカレート）
```

## ワークフロー

### フェーズ0: PR 前提チェック（自分）

```bash
gh pr view $ARGUMENTS --json number,state,headRefName,url
```

- `$ARGUMENTS` が空のときは `gh pr view`（引数なし）で**現ブランチに紐づく PR を自動検出**する。検出できた PR をそのまま対象として続行し、ユーザー確認は求めない。
- 自動検出にも失敗した場合のみ「PR 未特定」とみなして**停止**し、PR の指定／作成を促す（下記「フェーズ0 NG 時の報告」参照）。フェーズ1以降・サブエージェントは起動しない。ローカル未コミット差分を代わりの対象にしない。
- PR が closed/merged → ユーザーに確認してから判断する。
- OK なら現在のコミットを記録（`git rev-parse HEAD` → `LOOP_START_HEAD`）し、ループに入る。

#### フェーズ0 NG 時の報告

PR 前提を満たさず停止するときは、次を簡潔に伝える。

- 停止理由（PR 未特定 / ユーザー未確認の closed・merged 等）
- ユーザーアクション（PR 番号・URL・ブランチの指定、または PR 作成）
- ローカル未コミット差分は本コマンドの対象外である旨（差分があっても触らない）

### フェーズ1: レビュー（自分 = `/code-review` 基準）

Skill `code-review`（または slash command `/code-review`）を対象 PR に対して起動する。いずれも無い環境では、下記5観点の手順を自分で実行する。共通基準：5観点の並列レビュー（CLAUDE.md 準拠 / 明白なバグの浅いスキャン / git 履歴 / 過去PRコメント / コード内コメント準拠）→ 各指摘を 0-100 でスコアリング → **スコア80未満を除外**、偽陽性（既存issue・lint/型/テストで拾える系・スコープ外の行など）を除外。

- **対応すべき指摘 = スコア80以上の残存指摘**。
- **それがゼロ（= `/code-review` が "No issues found" 相当）なら、ループを抜けてフェーズ5（完了報告）へ。**
- 残っている場合は、各指摘を後段で扱えるよう構造化して保持する（指摘ID / 重大度・スコア / 対象ファイル:行 / 根拠 / 期待する修正方向）。

> `/code-review` は通常 PR へコメントする。ループ運用では、コメントは記録として残してよいが、**ループの分岐判定は「スコア80以上の指摘が残っているか否か」で行う**。

### フェーズ2: 指摘のグルーピング（自分）

並列起動の前に、残った指摘を**「対象ファイル集合が重ならない」グループ**に束ねる。これが競合回避の要。

1. 各指摘の対象ファイル集合を洗い出す。
2. ファイル集合が1つでも重なる指摘同士は**必ず同一グループ**（同一ファイルの同時編集は壊れる）。
3. ファイル集合が完全に独立した指摘は**別グループ**に分けられる。
4. 強い依存関係がある指摘（A の修正が B の前提等）は、ファイルが重ならなくても同一グループに束ね、グループ内で順次対応させる。
5. 「グループ間はファイル非競合かつ独立」が保証される状態にする。

成果物: グループ一覧（各グループに `担当ファイル集合` / `含む指摘 ID と内容` / 任意で `dependency`（例: R1→R2）/ 前周から同一指摘が残る場合は `prompt_notes`（強化方針・未達理由））。グループ数 = 並列起動する修正サブエージェント数（1グループのみのときは 0 または自分）。

> 全指摘が同一ファイル群に集中しグループが1つしかできない場合は、無理に分割せず1サブエージェント（または自分）で順次対応する（その旨を報告）。

### フェーズ3: 並列修正（サブエージェントをグループ数分・並列起動）

グループごとにサブエージェントを起動する。**1回のメッセージ内で全グループ分の Agent 呼び出しを並列発行する**（直列に1つずつ起動しない）。**各サブエージェントは担当指摘を自分でコード修正する。外部ツール（Codex 等）への委譲はしない。**

**グループが1つで修正量が小さい場合**は、サブエージェントを起動せず自分で直接修正し、1コミットにまとめてフェーズ4へ進んでよい。小さいの目安: 担当ファイルが1〜2個、指摘が3件以下、各修正が局所（おおむね数行以内）で依存が軽い。

グループが2つ以上あるときは、軽量でも**サブエージェントを並列起動**する（並列化の核心を優先）。

各サブエージェントへの prompt テンプレート:

```
あなたは「レビュー → 修正チーム」の修正サブエージェントです（グループ <番号>/<総数> 担当）。
担当指摘を自分でコード修正し、完了後にコミットします。Codex 等への委譲はしません。

## 修正の進め方
1. 下記「担当する指摘」を読み、対象コードを Serena（`get_symbols_overview` / `find_symbol` / `replace_symbol_body` 等）または通常のファイル編集で修正する。
2. 修正後、(a) 担当ファイル以外に触れていないか、(b) 指摘の範囲を超えていないか、(c) 指摘どおりに直っているか を自分で確認する。
3. グループ内に依存順がある場合はその順で対応する。

## 厳守事項
- 下記「担当ファイル」以外のファイルは絶対に編集しない（他サブエージェントと並列実行中。担当外への変更は競合を起こす）。
- 割り当てられた指摘だけを修正する。指摘の範囲を超える変更はしない。逸脱が必要なら理由を述べてから行う。

## コミット
- 担当分を **1コミット**にまとめる。`git add <担当ファイル>` で担当分だけをステージし（`git add -A` 禁止）、Conventional Commits 形式でコミットする（`git-ops` skill に従う）。コミット本文に対応した指摘 ID を列挙する。

## 担当ファイル
<このグループのファイル集合>

## 担当する指摘（重大度・スコアつき）
<このグループの指摘 ID・対象ファイル:行・根拠・期待する修正方向>

## 依存順・前周からの強化メモ（あれば）
dependency: <例: R1→R2>
prompt_notes: <前周未達理由・期待 diff・次周で強化する具体方針。初回周回なら空でよい>

## 報告
修正した指摘 ID、変更ファイル、コミットハッシュ、未対応があれば理由を返す。
```

サブエージェントの種類: 軽量な局所修正（単一ファイル・依存なし）は `generalPurpose`。**強い依存で束ねたグループ**や複数ファイル跨ぎは `task-executor`。

並列数が多い／修正が長時間化する場合はサブエージェントを `run_in_background: true` で起動し `TaskOutput` で回収してよい。

### フェーズ4: 修正コミットの確認（自分）

全サブエージェント（または自分の直接修正）の完了を待ってから確認する。比較基準は**当該周回の開始時点**の `LOOP_START_HEAD`（各周回のフェーズ3の直前に `git rev-parse HEAD` で更新し、当該周で生じたコミットだけを見る）。

```bash
git log --oneline $LOOP_START_HEAD..HEAD
git diff $LOOP_START_HEAD..HEAD --stat
```

- 各グループが意図通り1コミットになっているか。
- 担当外ファイルへの越境変更が混ざっていないか。
- 機械的な型・リント・テスト検証が必要なら `quality-gainner` subagent に委譲してよい。

確認できたら**フェーズ1へ戻り、再度 `/code-review` をかける**（ループ周回）。

### ループ判定と打ち切り

- **抜ける条件**: フェーズ1で `/code-review` の対応すべき指摘（スコア80以上）が**ゼロ**になったら、ループを抜けてフェーズ5（完了）。
- **周回上限**: 最大**3周**。3周してもスコア80以上の指摘が残る場合はループを打ち切り、フェーズ5で**ユーザーにエスカレート**する。
- 各周回で「新規指摘が増えていないか／同じ指摘が直っていないか」を観察する。同一指摘が直らない場合は指摘文・グループ分けの曖昧さを疑い、フェーズ2の `prompt_notes` に前周の未達理由・期待 diff・依存順を追記し、次周のサブエージェント prompt にそのまま埋め込む。無限往復しない。
- 再レビューで指摘の根拠が他指摘 ID を参照している場合は依存を再判定し、必要なら再束ねする（例: R1 解消後も R2 が「R1 追随不足」なら R2 単独でなく関連ファイルを再確認）。

### フェーズ5: 報告（自分）

プロセスではなく結論を簡潔に報告する。

- 結果: **OK 到達**（指摘ゼロでループ終了）か **3周到達でエスカレート**か。
- 周回数 / 各周の指摘件数の推移 / 最終的な未対応指摘（理由つき）。
- 各グループのコミットハッシュ。
- 仮置きした判断・ユーザー判断が必要な残課題。

## 成果物

- `/code-review` 基準のレビュー指摘リスト（周回ごと）
- 非競合グループ一覧（並列計画）
- 各サブエージェント（または自分）による修正コミット（グループごとに1コミット）
- ループ結果サマリ（OK到達 / エスカレート、周回推移、残課題）
