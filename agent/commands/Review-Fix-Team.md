# レビュー → 修正オーケストレーションチーム

## 目的

PR の差分を `/code-review`（Skill: `code-review`）基準でレビューし、その指摘を**ファイル単位の非競合グループに束ねてグループ数分の修正サブエージェントを並列起動**する。**各サブエージェントは自前でコードを書かず、Codex（`mcp__codex__codex` / `codex exec`）に修正を委譲し、その Codex の進行をモニター・回収してコミットする。** これを**再度 `/code-review` をかけて「対応すべき指摘がゼロ／OK」になるまでループ**する。Opus は司令塔として、レビューの起動・指摘のグルーピング・並列修正の統括・ループ判定に専念し、コードの書き換えはサブエージェント配下の Codex に委譲する。

## 前提（PR が必須）

レビュー基準として用いる `/code-review` は **GitHub PR を前提**とし、指摘を PR にコメントする設計である。本コマンドもこれに合わせ、**対象 PR がある場合のみ動作**する。

- `$ARGUMENTS` で対象 PR を指定する（PR 番号 / PR URL / PR ブランチ）。
- **PR が特定できない場合は、レビューを起動せずここで停止し、ユーザーに「PR を指定／作成してください」と伝える。** ローカルの未コミット差分だけを対象に本コマンドを動かさない。

> 前提スキル: 作業開始前に `task-orchestration` スキル（並列実行の原則：「同一ファイルの同時編集禁止」「依存タスクは順次」）と `opus-codex-orchestration` スキル（Opus 司令塔 ←→ Codex オペレーターの下請け構造、Codex 起動手段の使い分け）を読み、そのまま適用する。コミットは `git-ops` skill に従う。

## 役割分担

| 役割 | 担当 | 仕事 |
| ---- | ---- | ---- |
| 司令塔 / レビュー統括 | **Opus**（自分） | `/code-review` を起動して指摘を取得し、**ファイル単位の非競合グループ**に束ねる。グループ数分の修正サブエージェントを並列起動し、修正後に再レビューしてループ判定する。**コードは書かない** |
| 修正サブエージェント（複数・並列） | **サブエージェント**（グループ数分） | 割り当てられた**1グループの指摘だけ**を扱う。**自分でコードを書かず Codex に修正を委譲し、Codex の進行をモニター・回収**する。担当外ファイルには触れず、完了後に**自分で1コミット**する |
| 修正オペレーター | **Codex**（各サブエージェント配下） | サブエージェントから渡された指摘どおりに**実際のコード修正を実行**する。`mcp__codex__codex` / `mcp__codex__codex-reply` / `codex exec` で起動される |

## 並列化の核心

**1つのサブエージェントに全指摘を順番に直させない。** 指摘を非競合グループに分け、グループの数だけサブエージェントを**1回のメッセージ内で同時に**起動する（複数 Agent 呼び出しを並列発行）。各サブエージェントはさらに配下の Codex に修正を委譲し、その進行をモニターする。N グループが N 並列で同時に修正・コミットされ、直列実行の待ち時間を排除する。

## ループ概要

```
[PR前提チェック] → (NGなら停止)
   ↓
┌──────────── ループ（最大3周）────────────┐
│ フェーズ1: /code-review でレビュー（Opus）   │
│   → 対応すべき指摘がゼロ/OK なら ⇒ 完了で抜ける │
│ フェーズ2: 指摘のグルーピング（Opus）         │
│ フェーズ3: グループ数分サブエージェント並列起動 │
│   （各サブエージェントが Codex を起動・モニター）│
│ フェーズ4: 各修正のコミット確認（Opus）        │
└─────────── 周回して再びフェーズ1へ ──────────┘
   ↓
フェーズ5: 報告（OK到達 or 3周到達でエスカレート）
```

## ワークフロー

### フェーズ0: PR 前提チェック（Opus）

```bash
gh pr view $ARGUMENTS --json number,state,headRefName,url
```

- PR が特定できない → **停止**し、PR の指定／作成を促す（上記「前提」参照）。
- PR が closed/merged → ユーザーに確認してから判断する。
- OK なら現在のコミットを記録（`git rev-parse HEAD`）し、ループに入る。

### フェーズ1: レビュー（Opus = `/code-review` 基準）

Skill `code-review` を対象 PR に対して起動する。`/code-review` は内部で次を行う基準を持つ：5観点の並列レビュー（CLAUDE.md 準拠 / 明白なバグの浅いスキャン / git 履歴 / 過去PRコメント / コード内コメント準拠）→ 各指摘を 0-100 でスコアリング → **スコア80未満を除外**、偽陽性（既存issue・lint/型/テストで拾える系・スコープ外の行など）を除外。

- **対応すべき指摘 = スコア80以上の残存指摘**。
- **それがゼロ（= `/code-review` が "No issues found" 相当）なら、ループを抜けてフェーズ5（完了報告）へ。**
- 残っている場合は、各指摘を後段で扱えるよう構造化して保持する（指摘ID / 重大度・スコア / 対象ファイル:行 / 根拠 / 期待する修正方向）。

> `/code-review` は通常 PR へコメントする。ループ運用では、コメントは記録として残してよいが、**ループの分岐判定は「スコア80以上の指摘が残っているか否か」で行う**。

### フェーズ2: 指摘のグルーピング（Opus）

並列起動の前に、残った指摘を**「対象ファイル集合が重ならない」グループ**に束ねる。これが競合回避の要。

1. 各指摘の対象ファイル集合を洗い出す。
2. ファイル集合が1つでも重なる指摘同士は**必ず同一グループ**（同一ファイルの同時編集は壊れる）。
3. ファイル集合が完全に独立した指摘は**別グループ**に分けられる。
4. 強い依存関係がある指摘（A の修正が B の前提等）は、ファイルが重ならなくても同一グループに束ね、グループ内で順次対応させる。
5. 「グループ間はファイル非競合かつ独立」が保証される状態にする。

成果物: グループ一覧（各グループに `担当ファイル集合` / `含む指摘 ID と内容`）。グループ数 = 並列起動する修正サブエージェント数。

> 全指摘が同一ファイル群に集中しグループが1つしかできない場合は、無理に分割せず1サブエージェント（配下の Codex）で順次対応する（その旨を報告）。

### フェーズ3: 並列修正の委譲（サブエージェントをグループ数分・並列起動 → 各サブエージェントが Codex をモニター）

グループごとにサブエージェントを起動する。**1回のメッセージ内で全グループ分の Agent 呼び出しを並列発行する**（直列に1つずつ起動しない）。**各サブエージェントは自分でコードを書かず、Codex に修正を委譲してその進行をモニターする。Opus もコードを書かない。**

各サブエージェントへの prompt テンプレート:

```
あなたは「レビュー → 修正チーム」の修正サブエージェントです（グループ <番号>/<総数> 担当）。
あなた自身はコードを書きません。実際の修正は Codex に委譲し、その進行をモニター・回収してコミットします。

## 修正の進め方（Codex への委譲とモニター）
1. `mcp__codex__codex`（長め・対話継続が要るなら `mcp__codex__codex-reply`、軽量なら `codex exec`）で Codex を起動し、下記「担当する指摘」と「担当ファイル」「厳守事項」をそのまま渡して修正を実行させる。
2. Codex の出力をモニターし、(a) 担当ファイル以外に触れていないか、(b) 指摘の範囲を超えていないか、(c) 指摘どおりに直っているか を確認する。逸脱・取りこぼしがあれば `codex-reply` で差し戻して修正させる。
3. Codex は実行のみ。コミットはあなた（サブエージェント）が行う。

## 厳守事項（Codex にもそのまま伝える）
- 下記「担当ファイル」以外のファイルは絶対に編集しない（他サブエージェントと並列実行中。担当外への変更は競合を起こす）。
- 割り当てられた指摘だけを修正する。指摘の範囲を超える変更はしない。逸脱が必要なら理由を述べてから行う。
- グループ内に依存順がある場合はその順で対応する。

## コミット（サブエージェント自身が行う）
- Codex の修正をモニターで確認できたら、担当分を **1コミット**にまとめる。`git add <担当ファイル>` で担当分だけをステージし（`git add -A` 禁止）、Conventional Commits 形式でコミットする（`git-ops` skill に従う）。コミット本文に対応した指摘 ID を列挙する。

## 担当ファイル
<このグループのファイル集合>

## 担当する指摘（重大度・スコアつき）
<このグループの指摘 ID・対象ファイル:行・根拠・期待する修正方向>

## 報告
修正した指摘 ID、変更ファイル、コミットハッシュ、Codex に差し戻した回数、未対応があれば理由を返す。
```

並列数が多い／修正が長時間化する場合はサブエージェントを `run_in_background: true` で起動し `TaskOutput` で回収してよい。

### フェーズ4: 修正コミットの確認（Opus）

全サブエージェント（配下の Codex 修正を含む）の完了を待ってから確認する。

```bash
git log --oneline <ループ開始時のHEAD>..HEAD
git diff <ループ開始時のHEAD>..HEAD --stat
```

- 各グループが意図通り1コミットになっているか。
- 担当外ファイルへの越境変更が混ざっていないか。
- 機械的な型・リント・テスト検証が必要なら `quality-gainner` / `green-loop` skill に委譲してよい。

確認できたら**フェーズ1へ戻り、再度 `/code-review` をかける**（ループ周回）。

### ループ判定と打ち切り

- **抜ける条件**: フェーズ1で `/code-review` の対応すべき指摘（スコア80以上）が**ゼロ**になったら、ループを抜けてフェーズ5（完了）。
- **周回上限**: 最大**3周**。3周してもスコア80以上の指摘が残る場合はループを打ち切り、フェーズ5で**ユーザーにエスカレート**する。
- 各周回で「新規指摘が増えていないか／同じ指摘が直っていないか」を観察する。同一指摘が直らない場合は指摘文・グループ分けの曖昧さを疑い、次周の prompt を強化する。無限往復しない。

### フェーズ5: 報告（Opus）

プロセスではなく結論を簡潔に報告する。

- 結果: **OK 到達**（指摘ゼロでループ終了）か **3周到達でエスカレート**か。
- 周回数 / 各周の指摘件数の推移 / 最終的な未対応指摘（理由つき）。
- 各グループのコミットハッシュ（必要に応じて Codex への差し戻し回数も）。
- 仮置きした判断・ユーザー判断が必要な残課題。

## 成果物

- `/code-review` 基準のレビュー指摘リスト（周回ごと）
- 非競合グループ一覧（並列計画）
- 各サブエージェント（配下の Codex が修正）による修正コミット（グループごとに1コミット）
- ループ結果サマリ（OK到達 / エスカレート、周回推移、残課題）
